# System Conventions

Cross-cutting systems registry — the things multiple features must integrate
with. Starts empty on purpose; it fills itself as the project grows.

## Standing rules
1. **Walk this file BEFORE building** any page/feature and wire into every
   system that applies.
2. **Register AFTER building**: if a feature introduced something other features
   must integrate with (a shared control, hook, app-wide event, data fallback,
   a naming/sign convention), add it here in the SAME change. This is part of
   finishing the feature, not optional.

## Registered systems

| System | The rule | Where |
|---|---|---|
| **Enforcer wake-queue** | ANY "react to X happening" behavior is implemented as a lightweight callback/watcher that calls `wake.put(...)` on the `Enforcer` queue — never a new polling loop, never a new thread doing COM work, never COM calls inside a callback. The Enforcer thread is the single place that owns COM objects and re-asserts state; its 15 s `queue.get` timeout is the app's only sanctioned "poll". | `micguard.py` — `Enforcer.run`/`_enforce`, `_VolumeCallback`, `_DeviceCallback` |
| **Config = DEFAULT_CONFIG merge** | Every setting is a key in `DEFAULT_CONFIG`, persisted only via `save_config()`, surfaced in the settings window, applied live via `enforcer.reattach()`/`poke()`. No env vars, no second file, no registry-stored settings. The `DEFAULT_CONFIG \| file` merge is the migration mechanism. | `micguard.py` — `DEFAULT_CONFIG`, `load_config`/`save_config`; [Dynamic-Settings.md](Dynamic-Settings.md) |
| **Single-source version via release.ps1** | `VERSION` in `micguard.py` is the only authoritative version; `release.ps1` bumps it, mirrors `pyproject.toml`, builds, tags `vX.Y.Z`, and publishes the exe to GitHub Releases. Nothing else may change a version number or create a tag — installed copies compare release tags against `VERSION` to offer updates. | `micguard.py` (`VERSION`), `release.ps1`, [../RELEASING.md](../RELEASING.md) |
| **User-consent convention** | Actions that change the user's machine beyond enforcing their chosen mic state (updating the exe, uninstalling, anything destructive) ask via a topmost dialog (`App._dialog`) and fail open to a manual path (e.g. opening the releases page). Enforcement itself is the product and never asks. An explicit user click on a button that NAMES the action counts as the consent — the Settings row's "Update & restart" button (v1.10 `update_info`/`install_update`) installs on one click with no extra dialog, and the ARP Uninstall path uses a native topmost MessageBox because the webview stack isn't running there. | `micguard.py` — `_update_check`, `update_info`/`install_update` (settings Api), `uninstall_from_arp`, `_uninstall`, `_dialog` |
| **Window styling system** | EVERY window/dialog is a frameless pywebview (WebView2) window built from the HTML templates in `micguard.py`: shared `BASE_CSS` (shadcn/zinc tokens: `#09090b` bg, `#27272a` hairlines, white primary button, `#22c55e` green ONLY for on/active states) + `SETTINGS_HTML`/`DIALOG_HTML`/`MENU_HTML`/`ALERT_HTML`/`OSD_HTML`/`MIXER_HTML`. New UI = extend those templates; never a tkinter window, never a second UI library, never inline hardcoded colors outside the templates' shared tokens. Rules that ride along: `background_color="#09090b"` on every `create_window` (white-flash guard); yes/no + info prompts go through `App._dialog` (screen-centered); the settings window, tray menu, fallback alert, hotkey OSD, and (v1.6) mixer popup are ALL persistent hide/show singletons — only `_quit` destroys windows, because webview.start() returning IS app exit; js_api handlers CoInitialize before Core Audio calls. **No-activate singletons** (alert, OSD, and — v1.6 — the mixer popup via `MIXER_HTML`) additionally go through `App._show_noactivate` (`WS_EX_NOACTIVATE \| WS_EX_TOOLWINDOW` + `SW_SHOWNOACTIVATE`, so games/fullscreen apps never lose focus) and MUST be primed once via a real activating show/hide cycle — `App._prime_window`, run at GUI-loop start through `webview.start(func=self._prime_windows)` — or WebView2 composites solid black forever (see Architecture Gotchas). The mixer additionally: sizes to content (`document.body.scrollHeight`) before placement, and places itself bottom-center on the monitor the CURSOR is on (`_mixer_position`, via `MonitorFromPoint` — see the HMONITOR-restype gotcha), not the monitor the game window is on. **(v1.10, mixer input)** `MIXER_HTML` is the first no-activate popup that takes real input: it carries its own `js_api` bridge (`hover`/`set_volume`/`scroll`) so hovering, click-dragging, and scrolling a row reach Python without the popup ever stealing focus from a fullscreen game — gated by `mixer_hover_select`/`mixer_drag`/`mixer_scroll` (see Dynamic-Settings.md) and auto-hiding after `mixer_timeout` seconds (`0` = stay open). | `micguard.py` — `BASE_CSS`, `SETTINGS_HTML`, `DIALOG_HTML`, `MENU_HTML`, `ALERT_HTML`, `OSD_HTML`, `MIXER_HTML`, `_make_settings_window`, `_make_menu_window`, `_make_alert_window`, `_make_osd_window`, `_make_mixer_window`, `_show_noactivate`, `_prime_window`/`_prime_windows`, `open_settings`, `open_menu`, `toggle_mixer`, `_dialog` |
| **Fullscreen-safe popup placement** | ANY popup that can appear while a game is running gets its monitor from `popup_monitor_rect(cfg)` — never from `webview.screens`, the primary monitor, or a hardcoded rect. It returns `(rect | None, tried_same)`: `None` means SUPPRESS the popup (hotkey behavior must still work without it), and `tried_same=True` means the exclusive game's own monitor was chosen and the caller MUST arm `App._arm_fse_probe(*App._fse_probe_target(), hide, reshow=...)` so an incompatible game is auto-restored and learned into `cfg["fse_incompatible"]` (the mixer passes a reshow callback; OSD/alert pass none and stay hidden until their next trigger). The decision logic lives in the PURE `pick_popup_monitor` (pytest-covered) — extend it there, not with ad-hoc monitor checks at call sites. Modes come from `cfg["fullscreen_popups"]` (`auto`/`other`/`off`). | `micguard.py` — `popup_monitor_rect`, `pick_popup_monitor`, `exclusive_fullscreen_active`, `_arm_fse_probe`, `_fse_probe_target`; [Features/Fullscreen-Safe-Popups.md](Features/Fullscreen-Safe-Popups.md) |
| **Settings-scoped live audio** | Live feedback in the settings window (the level meter's 20 Hz poll, the MicMonitor "hear yourself" passthrough) may run ONLY while the window is visible and MUST be stopped by `App._settings_closing()` — this is the sanctioned exception to "no polling", scoped to an open UI. Anything that lets the user fiddle with volume live sets `enforcer.hold_volume = True` (capture flow only — see Enforcer wake-queue row) and clears it (+ `poke()`) when done, so enforcement never fights a preview. The monitor is in-app WASAPI passthrough; never toggle Windows' "Listen to this device" instead. Applies only to the CAPTURE flow's meter/hear-yourself target (the currently enforced mic); render-flow volume is unaffected by it. | `micguard.py` — `MicMonitor`, `App._start_meter`/`_stop_meter`/`_set_monitor`/`_settings_closing`, `Enforcer.hold_volume` |
| **Tray interaction contract** | Left-click tray = Settings; right-click tray = the themed `MENU_HTML` menu anchored at the cursor (bottom-left corner at pointer, flips near edges, auto-hides on blur). This is done by `_patch_tray_clicks` replacing pystray's `WM_NOTIFY` handler — new tray actions get a row in `MENU_HTML` + an `Api` method in `_make_menu_window` AND a matching item in the native pystray fallback menu (used only if the patch fails on a future pystray version). The menu's Profiles section makes its height dynamic (JS reports `document.body.scrollHeight` after `refreshMenu()`, Python resizes before the real-rect cursor anchoring runs) so it grows/shrinks with the active profile's device count. | `micguard.py` — `_patch_tray_clicks`, `open_menu`, `MENU_HTML`, `App.run` (fallback menu) |
| **Hotkey manager** | Global volume hotkeys are a single `HotkeyManager` thread doing `RegisterHotKey` + a blocking `GetMessageW` loop — NEVER a keyboard hook, NEVER a polling loop (same rule as the Enforcer wake-queue row, applied to input instead of audio events). New bindings are config rows (`cfg["hotkeys"]["bindings"]`), surfaced in the settings Hotkeys section, never hardcoded. Any enable/disable/rebind goes through `App._restart_hotkeys` — which tears down the whole running instance (`shutdown()` + `join`) and starts a fresh one, rather than mutating a live thread's registrations. Ships with `hotkeys.enabled = False` — see Architecture Gotchas for why (RegisterHotKey grabs the combo system-wide). **Targets (v1.6):** `system` (default render volume), `app:<exe>` (a specific session, boost-aware past 100%), `active` (the foreground window's session, boost-aware), `mixer` (toggles the mixer popup, always `step: 0`, no session/boost logic of its own). **(v1.9)** `profile:next` (cycle to the profile after `active_profile`, wrapping — `"next"` is reserved even over a profile literally named that) and `profile:<name>` (switch straight to a named profile); both always `step: 0` and resolve through the pure `resolve_profile_target(target, cfg)`. **ALL profile switches — tray menu or hotkey — MUST go through `App.set_profile(name)`; never inline a profile switch again.** It is the single place that persists `active_profile`, re-asserts enforcement, and records the history row, so it's also what guarantees exactly one `history` "profile" row per switch no matter which UI triggered it. **Ephemeral mixer keys rule:** while the mixer popup is visible, its bare digit/arrow/Esc/R keys are registered/unregistered ONLY on the `HotkeyManager` thread — `R` resets the currently selected row (including the system row and muted rows) to 100% — other threads call `set_mixer_keys(on)`, which posts `WM_APP_MIXER_ON`/`_OFF` rather than calling `RegisterHotKey` directly (RegisterHotKey is per-thread); registration is idempotent (a second ON while keys are already held is a no-op) to avoid orphaning duplicate registrations. | `micguard.py` — `HotkeyManager`, `parse_hotkey`, `adjust_system_volume`, `boosted_nudge`, `App._restart_hotkeys`, `App.toggle_mixer`, `set_mixer_keys`, `App.set_profile`, `next_profile`, `resolve_profile_target` |
| **Optional extension card** | A feature that needs an external, user-installed dependency MicGuard will never bundle (exe-size Rule 1) gets an ALWAYS-VISIBLE settings card, never a hidden/gated section: (1) the card **advertises itself** — a not-installed state shows an explainer of exactly what the extension adds plus a single guided-setup button, no sliders/controls until the dependency is present; (2) it **self-detects state on every Settings open** (never cached across launches) — install/uninstall of the dependency outside MicGuard is picked up automatically, no stale state; (3) setup is a **guided, consent-based flow**: one dialog states what will be downloaded/from where/what steps follow, download+launch is automated (stdlib `urllib` + `os.startfile`), and every failure path falls back to opening the dependency's own website rather than stranding the user — mirrors the update-flow "User-consent convention" row above; (4) the writer/glue code that talks to the external dependency **logs and degrades** (Rule 5) — never raises into the UI, returns an empty string on success or a short user-facing error string on failure. First instance: Mic EQ (Equalizer APO). Any future extension (e.g. noise suppression) follows this exact shape. | `micguard.py` — `App._mic_eq_state`, `App._setup_mic_eq`, `App._apply_mic_eq`, `apo_config_dir`, `write_eq_config`; [Features/Mic-EQ-Extension.md](Features/Mic-EQ-Extension.md) |
| **Notable-event history** | When a feature does something user-visible and infrequent (a device switch, a self-heal, a settings save, a setup completion), record it via `app.history.add(kind, text)` in the SAME change that adds the behavior — this is part of finishing the feature, not optional, same spirit as this registry itself. NEVER record per-enforcement-pass noise (volume-hold snap-backs, mute re-asserts, watchdog passes, anything that fires every enforce cycle rather than on a state *change*) — that's an explicit, deliberate exclusion, not a gap to "fix" by adding more calls. Identical `kind`+`text` within the last 8 entries and 10 minutes coalesce into one row with a growing `×N` (the matched row moves back to the top) rather than spamming — lean on that instead of hand-rolling de-dup logic at the call site. `history.json` is a data file, not a config surface (see Dynamic-Settings.md) — there is no toggle to gate a new call site behind. | `micguard.py` — `history_push`, `HistoryRecorder`, `App.history`; [Features/Event-History.md](Features/Event-History.md) |
